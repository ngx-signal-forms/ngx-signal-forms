# Migration to v1.0.0-rc.11

## Upgrade from v1.0.0-rc.10

This release finalizes the public API and fixes correctness and accessibility
issues discovered during the v1.0 release audit.

### Required changes

- Add `@angular/common@">=22.0.0 <23.0.0"` to direct dependencies if your
  package manager does not already install it through Angular. It is now a
  declared toolkit peer dependency because the form-field entry point uses
  `NgComponentOutlet` and `NgTemplateOutlet`.
- Replace the form-level `errorStrategy="inherit"` value with the explicit
  strategy you want (`'on-touch'`, `'on-submit'`, or `'immediate'`). The
  form-level `NgxSignalForm` directive no longer accepts `'inherit'`.
- Replace removed notification tone APIs. `NgxNotificationTone`,
  `NgxFormFieldNotificationTone`, and the `tone` input on
  `NgxHeadlessNotification` / `NgxFormFieldNotification` are removed.
  Notification severity is now derived from the supplied messages.
- Update direct implementations of `ErrorSummarySignals` to include
  `shouldShowWarnings` and `resolvedStrategy`. Both are now part of that
  public interface.
- Update direct reads of `NgxHeadlessErrorState` and `createErrorState()`
  results: `showErrors` / `showWarnings` are now `shouldShowErrors` /
  `shouldShowWarnings`.
- Update direct implementations of `FieldsetStateSignals` to include
  `resolvedErrors`, `resolvedWarnings`, and `resolvedWarningStrategy`.
- Update `NgxFormMarkingLegend` root bindings from `[formField]` to
  `[formTree]`.
- Update public fieldset option type imports to the `NgxFormFieldset*` names:
  `NgxFormFieldsetAppearance`, `NgxFormFieldsetFeedbackAppearance`,
  `NgxFormFieldsetSurfaceTone`, and `NgxFormFieldsetValidationSurface`.
- `CharacterCountResult` members now use Angular's `Signal<T>` type and the
  result includes `hasLimit`. Update direct interface implementations and
  wrappers accordingly.

### Template migrations

```html
<!-- Before -->
<form [formRoot]="form" ngxSignalForm [errorStrategy]="'inherit'"></form>
<ngx-form-marking-legend [formField]="form" />
<ngx-form-field-notification tone="warning" [errors]="errors" />
<ngx-form-fieldset [fieldsetField]="form.address"></ngx-form-fieldset>

<!-- After -->
<form [formRoot]="form" ngxSignalForm [errorStrategy]="'on-touch'"></form>
<ngx-form-marking-legend [formTree]="form" />
<ngx-form-field-notification [errors]="errors" />
<ngx-form-fieldset [field]="form.address"></ngx-form-fieldset>
```

Choose the explicit form-level error strategy appropriate for the form.
Notifications derive their severity from the errors; do not replace `tone`
with a different input.

### Type migration

| Before                                                 | After                                                   |
| ------------------------------------------------------ | ------------------------------------------------------- |
| `NgxNotificationTone` / `NgxFormFieldNotificationTone` | Remove the import; notification tone is content-driven. |
| `NgxFieldsetAppearance`                                | `NgxFormFieldsetAppearance`                             |
| `NgxFieldsetFeedbackAppearance`                        | `NgxFormFieldsetFeedbackAppearance`                     |
| `NgxFieldsetSurfaceTone`                               | `NgxFormFieldsetSurfaceTone`                            |
| `NgxFieldsetValidationSurface`                         | `NgxFormFieldsetValidationSurface`                      |
| `showErrors` / `showWarnings` on `ErrorStateResult`    | `shouldShowErrors` / `shouldShowWarnings`               |

### Behavior changes to review

- `canSubmitWithWarnings()` now includes blocking errors on descendant fields.
  Forms that compensated for the prior root-only check can remove that
  workaround.
- `injectFieldControl()` now throws when a resolved id does not point to a
  runtime `FieldTree`. Ensure test doubles and custom control bindings expose
  a real field tree.
- `NgxFormFieldErrorSummary` only auto-focuses for the resolved `'on-submit'`
  strategy. Touch and immediate validation no longer move focus.
- Empty assistive live regions remain mounted and exposed. Update tests to
  assert their empty content rather than expecting no alert or status element.
- Class-based dark-mode integrations must override the public
  `--ngx-signal-form-*` tokens; the unsupported `:host-context(.dark)`
  heuristic is gone.
- A shared Vest suite used by concurrently-mounted independent field trees now
  serializes whole-suite validation. The later tree can wait for the earlier
  tree's async run instead of observing its result.
- Field, wrapper, and fieldset warnings now default to independent
  `'immediate'` timing. Set `warningStrategy="inherit"` where the previous
  inherited error timing is intentional.
- An explicitly bound `fields=[]` on `NgxHeadlessFieldset` now means
  “aggregate nothing”; it no longer falls back to the fieldset's own errors.
- Standalone headless error surfaces now honor
  `provideNgxSignalFormsConfig({ defaultErrorStrategy })`. Scope the provider
  with `provideNgxSignalFormsConfigForComponent()` when a standalone surface
  needs different timing.

### Additions and compatibility

- `@ngx-signal-forms/toolkit/testing` is now a supported secondary entry
  point. Its `axe-core` peer dependency is optional and only needed when that
  entry point is imported.
- `requiredFromStandardSchema()` is available from the root entry point for
  Standard Schema requiredness and auto-ARIA.
- The Vest peer range now accepts all supported 6.x versions.

### Complete the upgrade

Search the consumer codebase for every removed or renamed API in this guide,
then run its existing type-check, build, and relevant tests. The upgrade is
complete only when every required change is applied or confirmed inapplicable.

See [`MIGRATING_BETA_TO_V1.md`](../MIGRATING_BETA_TO_V1.md) for the complete
beta-to-v1 migration guide. Future releases must add their versioned migration
guide and an upgrade section before publishing.
