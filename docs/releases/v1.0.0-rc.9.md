# v1.0.0-rc.9

> ⚠️ **Breaking changes** — this RC closes out the v1 architecture deepening from PRD [#36](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/36) ([PR #37](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/37)). Three small public-API removals (`walkFieldTree`, `walkFieldTreeIterable`, `NgxFormFieldAssistiveRow`) plus a renderer-token seam that lets third-party wrappers swap `NgxFormFieldError` / `NgxFormFieldHint` without forking the wrapper. The default rendering for zero-config consumers is unchanged.
>
> RC.9 also lands the **v1 wrapper-author surface** in `@ngx-signal-forms/toolkit/headless`: pure ARIA signal factories (`createAriaInvalidSignal`, `createAriaRequiredSignal`, `createAriaDescribedBySignal`, `createHintIdsSignal`), reactive error resolution (`createErrorMessageSignal`), plus four wrapper-author helpers (`createAriaDescribedByBridge`, `createFieldNameResolver`, `toHintDescriptors`, `createErrorRendererInputs`). Three reference wrappers — **Angular Material 21+**, **Spartan**, and **PrimeNG** — ship as runnable demos and dogfood every primitive end-to-end.

## `@ngx-signal-forms/toolkit`

### Features

- **`feat(toolkit)!`: v1 final deepening — close wrapper/assistive seam, tighten public surface** ([#37](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/37), PRD [#36](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/36))
  - **Promote `NGX_SIGNAL_FORM_HINT_REGISTRY` to public API** — `NgxSignalFormHintRegistry` and `NgxSignalFormHintDescriptor` flip from `@internal` to `@public`. Third-party wrappers can now provide the registry to participate in auto-aria's `aria-describedby` chaining without depending on internal symbols. Runtime contract unchanged.
  - **Renderer tokens** — new `NGX_FORM_FIELD_ERROR_RENDERER` and `NGX_FORM_FIELD_HINT_RENDERER` injection tokens, plus `provideFormFieldErrorRenderer()` / `provideFormFieldHintRenderer()` in both environment-scope (`EnvironmentProviders`) and component-scope (`Provider[]`) flavours. Both look through to the parent injector via `{ optional: true, skipSelf: true }` so nested overrides cascade.
  - **`NgxFormFieldWrapper` + `NgxFormFieldset` consume the renderer tokens** — the wrapper no longer hard-instantiates `NgxFormFieldError`; it resolves the configured renderer via `*ngComponentOutlet`. Default fallback to `NgxFormFieldError` / `NgxFormFieldHint`, so zero-config consumers see no change. Fieldset binds the same token with five renderer inputs (`errors`, `fieldName`, `strategy`, `submittedStatus`, `listStyle`); custom error renderers used in fieldsets must accept those names.
  - **Public `isFieldTree()` predicate** — exposed shared `FieldTree` runtime contract check. Replaces ad-hoc duck-typing in consumer code (debugger, submission-helpers, walker now share one predicate).
  - **`NgxFormFieldAssistiveRow` inlined into `NgxFormFieldWrapper`** — the 30-line component had no external consumers and added a shallow public-API surface. CSS contract preserved: the same `--ngx-form-field-assistive-*` custom properties keep working at the wrapper scope.

- **`feat(headless)`: pure ARIA signal-factory primitives** ([#51](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/51), [#52](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/52), [#53](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/53), [#55](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/55), PRD [#38](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/38)) — four pure factories extracted from `NgxSignalFormAutoAria` so wrapper authors can compose ARIA without inheriting the directive shell. Each factory is `fieldState in → ARIA out` with no DI of its own; the directive itself now delegates to them so the public seam and the directive can never drift.
  - **`createHintIdsSignal()`** — pulls hint IDs from the optional shared `NgxFieldIdentity` service or the `NgxSignalFormHintRegistry` (with field-name filter) and yields the ordered ID list.
  - **`createAriaInvalidSignal()`** — takes `Signal<FieldState | null>` plus the visibility computed from `createErrorVisibility` and an optional `isControlVisible` predicate, returns `Signal<'true' | 'false' | null>`.
  - **`createAriaRequiredSignal()`** — takes a field state and yields `Signal<'true' | null>` matching the directive's behaviour.
  - **`createAriaDescribedBySignal()`** — composes hint IDs + error / warning IDs into a single `aria-describedby` value, with an optional `preservedIds` reader that lets host design systems keep their existing `aria-describedby` chain (Material's `mat-form-field` ownership, Spartan's `BrnField`).
  - All four sources live in `packages/toolkit/core/utilities/aria/` (to avoid circular secondary-entry-point dependencies) and are re-exported from `@ngx-signal-forms/toolkit/headless` (the documented public surface for these primitives).

- **`feat(headless)`: `createErrorMessageSignal()` reactive error-message resolution** ([#63](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/63), closes [#39](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/39), closes [#44](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/44)) — public primitive that composes `createErrorVisibility` + `resolveValidationErrorMessage` + `generateErrorId` into one reactive `Signal<readonly ResolvedFieldError[]>`. i18n consumers stop reaching across `/headless` and `/core` to glue these together by hand.
  - Auto-injects `NGX_ERROR_MESSAGES`; `options.errorMessages` overrides DI for tests / explicit-injector call sites.
  - Visibility-filtered output (single source of truth).
  - `includeWarnings: false | true | 'only'` selects which slice surfaces.
  - `stripWarningPrefix` defaults to `true` (display-oriented divergence from the underlying resolver's debugging-oriented default).
  - `generateErrorId(fieldName, kind?)` gains an optional `kind` parameter so per-error IDs interoperate between in-tree and external consumers.
  - **Flattened `ResolvedFieldError` shape**: `{ kind, message, id, error }`. Templates write `entry.kind` directly instead of the nested `entry.error.kind`; the raw `ValidationError` is still available on `entry.error` for consumers that need validator params or non-stripped messages.
  - **`NgxFormFieldError` migrates to the new primitive** for both blocking and warning resolution. `errorsOverride` now wins over `formField` in both visibility and message resolution (previously visibility short-circuited to override mode while messages still resolved from `formField`, leaving the alert container visible with empty content).
  - **`@for` track keys hardened** — track expressions in the live regions now use a template-literal key derived from `entry.kind` and `$index` instead of `entry.message`, so registry / i18n swaps no longer replace DOM nodes inside `role="alert"` / `role="status"` containers and risk duplicate AT announcements.

- **`feat(headless)`: four wrapper-author primitives** ([#57](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/57)) — hoists boilerplate every form-field wrapper otherwise reimplements into shared, unit-tested primitives. Motivated by the Spartan, Material, and PrimeNG reference wrappers reproducing the same logic with subtle drift.
  - **`createAriaDescribedByBridge({ toolkit })`** + the `AriaDescribedByBridge` interface — for design systems that own `aria-describedby` via an injectable a11y service (Spartan brain's `BrnFieldA11yService`, or any equivalent shape). Wrapper authors register the bridge with `useFactory` at the wrapper's component-level injector and let the design system's host binding write the toolkit-managed IDs onto the bound control. No more ~100-line bespoke bridges per integration.
  - **`createFieldNameResolver({ explicit, labelFor?, boundControl, wrapperName })`** — the canonical priority cascade: `explicit → labelFor → boundControl.id → null + one-shot dev warning`. Mirrors what `NgxFormFieldWrapper` does so every reference wrapper stays on the same resolver.
  - **`toHintDescriptors(hints)`** — maps a `Signal<readonly HintLike[]>` (typically `contentChildren(NgxFormFieldHint)`) to the wire format consumed by `NGX_SIGNAL_FORM_HINT_REGISTRY` and `createHintIdsSignal`. Structurally typed via `HintLike` so `/core` does not depend on `/assistive`. Adopted by `NgxFormFieldWrapper` itself in this RC so the canonical wrapper can never drift.
  - **`createErrorRendererInputs({ formField, strategy, submittedStatus })`** alongside the named `NgxFormFieldErrorRendererInputs<T>` type — builds the `*ngComponentOutlet` `inputs:` map for renderer-token-resolved error components. Names the renderer contract as a public type so consumers writing custom renderers see the same shape.

- **`feat(toolkit)`: dispatch `<ngx-form-field-hint>` via `NGX_FORM_FIELD_HINT_RENDERER`** ([#64](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/64), closes [#61](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/61)) — `NGX_FORM_FIELD_HINT_RENDERER` was advertised as part of the wrapper contract in RC.8 but `NgxFormFieldHint` ignored it (the demo wrappers' `provideFormFieldHintRenderer` registrations were no-ops). The component now lifts its projected children into the configured renderer after first render and forwards `resolvedFieldName`, `resolvedId`, `position` as inputs. Falls back to direct `<ng-content />` projection when no renderer is registered, so consumers using `<ngx-form-field-hint>` outside a wrapper see no change.

- **`feat(toolkit)`: `NgxSignalFormControlSemanticsDirective` exposes `elementRef`** ([#59](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/59), ADR-0002 §6) — the directive now publishes its host `ElementRef` for consumer wrappers that want to query the bound control via `contentChildren(NgxSignalFormControlSemanticsDirective)`. Used by the Material and Spartan reference wrappers for tier-3 field-name resolution without explicit `[fieldName]` props.

### Fixes

- **`fix(toolkit)`: resolve 5 correctness bugs** ([#73](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/73), PR [#74](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/74)) — five independent correctness bugs caught during a follow-up audit of the headless + submission surfaces. Regression tests added for each.
  1. **`submission-helpers`**: `submitWithWarnings` now adds the same `pending()` guard already present in `canSubmitWithWarnings()`, so async validators still in flight after the `queueMicrotask` delay cannot let the action fire on a stale-valid snapshot.
  2. **`headless/utilities`**: `createErrorState` now calls `injectFormContext()` at factory call time and resolves strategy / submittedStatus through `resolveStrategyFromContext` / `resolveSubmittedStatusFromContext`. Previously the factory always fell back to `'on-touch'` even inside an on-submit form. JSDoc corrected to document the precedence (option override → form context → default) and the `NgxSignalForm` host directive as the context provider (the previous wording wrongly attributed it to `provideNgxSignalFormsConfig`).
  3. **`character-count`**: `NgxHeadlessCharacterCount` gains a one-shot `#warnedUnsupportedValue` guard and `console.warn` for unsupported value types, matching the existing warning in the `createCharacterCount` factory. **Logs a constructor-name / `typeof` descriptor only** — never the raw user-entered value — to avoid surfacing user data in dev consoles, CI logs, and screenshots.
  4. **`normalize-config`**: JSDoc clarifies that `normalizeSignalFormsConfig` is a two-tier utility only (no DI parent-scope inheritance), documenting the intentional divergence from `provideNgxSignalFormsConfig`.
  5. **`form-field-wrapper`**: removes the synchronous `findBoundControl` / `requireHostElement` fallback inside the `resolvedFieldName` `computed()`. `requireHostElement` throws `TypeError` in SSR when `nativeElement` is not `HTMLElement`. The reactive `afterEveryRender` signal (`#inputElementId`) is the correct and only fallback path.

- **`fix(toolkit)`: post-merge review follow-ups** ([#60](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/60)) — type the `*ngComponentOutlet` inputs object against `NgxFormFieldErrorRendererInputs` so wrapper and renderer contracts stay in sync (contract drift becomes a compile-time error in the shared helper). Preserves the existing `Record<string, unknown>` return shape.

- **`fix(toolkit)`: `errorsOverride` over `formField` in resolved-errors accessor** ([#63](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/63)) — when both `[formField]` and `[errors]` are bound, the headless directive's `showErrors` short-circuits to override mode, but the resolved-errors accessor used to read `formField` first. Inverted the order so override wins in both visibility and message resolution. Regression test covers the dual-bind case.

### Refactoring

- **`refactor(toolkit)!`: drop `walkFieldTree` and `walkFieldTreeIterable`** ([#37](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/37)) — `walkFieldTreeEntries` is the canonical primitive. The other two were trivially derived from it and forced consumers to choose between three near-identical exports. All internal callers (`focus-first-invalid`, `submission-helpers`, `signal-form-debugger`) migrated. `walkFieldTreeEntries` additionally yields a stable dotted path per entry — useful for tracking, debugging, and structured logging.

  ```diff
  - walkFieldTree(form, (state) => { ... });
  - for (const state of walkFieldTreeIterable(form)) { ... }
  + for (const { state } of walkFieldTreeEntries(form)) { ... }
  ```

- **`refactor(toolkit-headless)`: dedupe 3-tier error message resolution** ([#67](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/67), ADR-0003) — extracts the shared `resolveValidationErrorMessage(error, registry, { stripWarningPrefix: true })` call into a private `resolveErrorMessage` helper in `utilities.ts`, so `NgxHeadlessErrorState` and `createErrorMessageSignal` are guaranteed to stay in lockstep. ADR-0003 records the decision and the rejected delegation alternative.

- **`refactor(toolkit/core)`: share `FieldTree` contract checks** ([#37](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/37)) — extract the runtime `FieldState` contract into a shared internal utility; reuse the predicate and required-method list across submission helpers and the field-tree walker so the validation contract lives in one place.

### ⚠️ Breaking changes summary

| Area              | Change                                                                                      | Action                                                                                      |
| ----------------- | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| Field-tree walker | `walkFieldTree` and `walkFieldTreeIterable` removed                                         | Migrate to `walkFieldTreeEntries` (yields `{ state, path }` tuples)                         |
| Assistive row     | `NgxFormFieldAssistiveRow` no longer exported                                               | Use the wrapper's inline layout, or roll your own (the original was 30 lines)               |
| Error renderer    | `NgxFormFieldWrapper` and `NgxFormFieldset` resolve through `NGX_FORM_FIELD_ERROR_RENDERER` | Consumers swapping to a custom renderer must call `provideFormFieldErrorRenderer({...})`    |
| Hint renderer     | `NgxFormFieldHint` now dispatches through `NGX_FORM_FIELD_HINT_RENDERER` when registered    | None — falls back to `<ng-content/>` when no renderer is registered (zero-config unchanged) |

CSS classes (`.ngx-signal-form-field-wrapper__*`, `.ngx-signal-form-fieldset--invalid`) and data attributes (`data-ngx-signal-form-control-*`) are intentionally unchanged.

### `apps/demo-material` — Angular Material 21+ reference wrapper (NEW)

> [#59](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/59), [#69](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/69), [#71](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/71)

The first of three reference wrapper integrations under PRD #40. Demonstrates the toolkit's public seam (renderer tokens, control semantics directive, four ARIA primitive factories) composed on top of `mat-form-field` / `mat-error` / `mat-hint` without forking the toolkit.

- **Key design choice** — the wrapper is a **directive applied directly on `<mat-form-field>`** (not a wrapping component) so Material's lexical `contentChildren(MatFormFieldControl)` query still sees the projected `<input matInput>` / `<mat-select>`.
- **`aria-describedby` interop** — Material owns `aria-describedby` on the bound control. The consumer opts the toolkit out via `ngxSignalFormControlAria="manual"` and `createAriaDescribedBySignal` uses its `preservedIds` reader to layer toolkit-managed IDs on top of Material's existing chain rather than overwriting it.
- **Primitives consumed** — the wrapper dogfoods every public toolkit primitive shipped in this RC: `createAriaDescribedBySignal`, `createAriaInvalidSignal`, `createAriaRequiredSignal`, `createHintIdsSignal`, `createFieldNameResolver`, `toHintDescriptors`, `createErrorRendererInputs`, and `createErrorMessageSignal`. The contact-form template loses the per-field `*ngComponentOutlet`, the `feedbackRenderer` field, the `rendererInputs` helper, and the `inject(NGX_FORM_FIELD_ERROR_RENDERER)` call.
- **`NgxMatFeedbackOutlet`** — drop-in component that injects `NGX_FORM_FIELD_ERROR_RENDERER` and mounts the configured renderer for `{ message, severity }`. Renderer swap via `provideNgxMatForms({ feedbackRenderer })` keeps working without any consumer-side wiring.
- **`NgxMatFormBundle`** — single-symbol consumer import covering the wrapper, the error renderer, and `NgxSignalFormControlSemanticsDirective`. Forward-compatible with a future `@ngx-signal-forms/material` package.
- One representative form (text input + select + checkbox) with a `kind: 'warn:short-name'` warning rendering inside `<mat-hint>` (separate from `<mat-error>`).
- Vitest smoke spec asserts invalid → `<mat-error>` content visible, `aria-invalid="true"`, `aria-describedby` resolves to the `<mat-error>` element. Pinning spec for `*ngxMatFeedback` behaviour via `createErrorMessageSignal`.
- Single Playwright spec covers fill → blur → observe-error in a real browser.
- App pins `@angular/material@~21.2.9`, `@angular/cdk@~21.2.9`, and `@angular/animations@21.2.10` in its **own** `package.json`. The toolkit's `package.json` is untouched; `pnpm check:toolkit-peer-deps` and the `toolkit-isolation` CI job both stay green.

### `apps/demo-spartan` — Spartan reference wrapper (NEW)

> [#57](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/57), [#70](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/70)

Canonical "host directive" reference under PRD #40 / ADR-0002. Composes the toolkit's renderer / hint / context seam on top of Spartan's `@spartan-ng/brain` `BrnField` host directive.

- **`SpartanFormFieldComponent`** declares `BrnField` as a host directive on the wrapper itself and provides `NGX_SIGNAL_FORM_FIELD_CONTEXT` + `NGX_SIGNAL_FORM_HINT_REGISTRY` so projected hints chain through auto-aria. The error slot resolves through the renderer token with `NgxSpartanFormFieldError` as the wrapper-owned default.
- **`aria-describedby` interop** — `[hlmInput]` declares `BrnFieldControlDescribedBy` as a host directive, which owns `aria-describedby` on the helm input host element. To prevent the DOM tug-of-war between Brain's host binding and the toolkit's auto-aria writes, the wrapper provides an inline `BrnFieldA11yService` factory whose `describedBy` signal mirrors the toolkit composition (hint IDs + error/warning IDs gated on the strategy). `BrnFieldControlDescribedBy` then writes the toolkit-managed IDs onto the helm input through its **own** host binding — no overwrites, no missed announcements.
- **Selector tightening** — the wrapper's `formField` input is aliased to `ngxSpartanFormField` and the selector tightened to `spartan-form-field[ngxSpartanFormField]`. Prevents Angular Signal Forms' `FormField` directive (selector `[formField]`) from double-binding to the wrapper's host element when consumers import both into the same template.
- **`NgxSpartanFormBundle`** — single-symbol consumer import mirroring `NgxMatFormBundle`. Forward-compatible with a future `@ngx-signal-forms/spartan` package.
- Migrated to `createErrorMessageSignal` ([#70](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/70)) alongside the four wrapper-author primitives so the bridge file shrinks from ~117 to ~88 lines.
- One representative form (text input + select + checkbox) with warnings on the `displayName` field via `validate(..., (v) => ({ kind: 'warn:short-display-name', ... }))`.
- Vitest smoke spec + Playwright spec covering fill → blur → observe-error and `aria-describedby` round-trip through Brain's host binding.
- App pins `@spartan-ng/brain@0.0.1-alpha.683` via the pnpm catalog.
- New **`libs/ui`** workspace library — Spartan-CLI-scaffolded `@spartan-ng/helm` components (input, label, select, checkbox) consumed in-tree via tsconfig path aliases (no built or published artifact). README documents the upstream-fidelity policy.

## Breaking-change cheatsheet (RC.8 → RC.9)

| Old                                            | New                                                            |
| ---------------------------------------------- | -------------------------------------------------------------- |
| `walkFieldTree(form, visitor)`                 | `for (const { state } of walkFieldTreeEntries(form)) { ... }`  |
| `walkFieldTreeIterable(form)`                  | `walkFieldTreeEntries(form)` (yields `{ state, path }` tuples) |
| `import { NgxFormFieldAssistiveRow } from ...` | (removed — wrapper inlines the layout; CSS tokens unchanged)   |
| Custom `NgxFormFieldError` via subclass / fork | `provideFormFieldErrorRenderer({ component: MyRenderer })`     |
| `entry.error.kind` in `ResolvedFieldError`     | `entry.kind` (raw `ValidationError` still on `entry.error`)    |

The v1 breaking-change surface for default consumers is intentionally narrow: replace the two `walkFieldTree*` calls if you have them, drop any direct import of `NgxFormFieldAssistiveRow`, and you're current. Custom-renderer migration is opt-in and only affects consumers who were already forking `NgxFormFieldError`.
