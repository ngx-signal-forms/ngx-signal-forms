# v1.0.0-rc.7

> ⚠️ **Breaking changes** — this RC closes out the v1 architecture triage from
> [#28](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/28).
> Public-API signatures are unchanged, but the **CSS theming surface** was
> consolidated (token renames + the SCSS → native-CSS shipping format) and
> **fieldset defaults** flipped to favour the new grouped-notification
> appearance. See [`docs/MIGRATING_CSS_VARS.md`](../MIGRATING_CSS_VARS.md) and
> the new fieldset section in [`docs/MIGRATING_BETA_TO_V1.md`](../MIGRATING_BETA_TO_V1.md#fieldset-component--headless-directive)
> for the full before/after tables.
>
> All five follow-up PRDs from the v1 architecture triage landed
> (#23, #24, #25, #26, #27). RC.7 is intended as the **final RC before
> the `1.0.0` stable cut** — no further architectural scope is planned.

## `@ngx-signal-forms/toolkit`

### ✨ Features

- **`feat(toolkit)!`: grouped notifications and adaptive fieldset feedback**
  ([#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22))
  - **New styled component: `NgxFormFieldNotification`** — a surfaced
    notification card for grouped error/warning summaries on a fieldset.
    Owns its own `role="alert"` / `role="status"` live regions (split into
    dual stable shells so the role attribute never re-assigns at the same
    tick as content insertion, matching `NgxFormFieldError`'s WCAG 4.1.3
    pattern). Inputs: `errors`, `submittedStatus`, `tone`, `listStyle`,
    `notificationTitle`. `tone='warning'` no longer escalates to
    `role='alert'` for warning-only message lists; blocking errors keep
    alert semantics regardless of the requested tone (content wins).
  - **New fieldset inputs:** `feedbackAppearance` (`'plain'` | `'notification'`),
    `surfaceTone` (`'auto'` | `'invalid'` | `'warning'` | `'neutral'`),
    `validationSurface` (`'never'` | `'always'`), `notificationTitle`,
    `listStyle`. One-shot dev-mode warnings fire when these are passed
    against `feedbackAppearance="plain"`.
  - **Adaptive feedback on `NgxFormFieldWrapper` for selection clusters**
    (radio groups, checkbox groups): grouped feedback now lives on the
    wrapper with a neutral projected heading slot (no fake `<label>`).
    Progressive notification transitions keep grouped feedback shells
    mounted across show/hide for smoother enter/exit animation.
  - **Custom-element fieldset is now a labelled group:**
    `<ngx-form-fieldset>` / `[ngxFormFieldset]` apply `role="group"` and
    `aria-labelledby="<fieldsetId>-legend"` only when an accessible name
    exists (projected `<legend>`, `aria-labelledby`, or `aria-label`),
    avoiding unnamed groups in the AT tree. Native `<fieldset>` hosts
    keep their intrinsic semantics untouched.
  - **`prefers-contrast: more`** and **`forced-colors: active`** blocks on
    the notification card so high-contrast and Windows forced-colors
    users keep a semantic error/warning border.

- **`feat(headless)`: `NgxHeadlessNotification` directive**
  ([#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22),
  [#30](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/30)) —
  renderless headless primitive backing `NgxFormFieldNotification`.
  Exposes `NotificationStateSignals`, `NgxNotificationTone`,
  `ResolvedNotificationMessage`. Added to the `NgxHeadlessToolkit` bundle.

- **`feat(headless)`: compose `NgxFormFieldError` over `NgxHeadlessErrorState`**
  ([#30](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/23),
  PR [#30](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/30)) —
  the styled error component is now a thin shell over the headless
  directive via `hostDirectives`, with a `connectFieldState()` bridge to
  forward field state without colliding with Angular's `FormField`
  directive selector. Removes ~150 lines of duplicated logic
  (`resolveStrategyFromContext`, `resolveSubmittedStatusFromContext`,
  `createShowErrorsComputed`, `splitByKind`, `resolveValidationErrorMessage`).
  Cross-surface specs prove same-tick visibility parity by construction.

- **`feat(toolkit)`: cascading-resolver primitive**
  ([#25](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/25),
  PR [#31](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/31)) —
  internal `createCascadingResolver<T>()` collapses every `?? parent ?? DEFAULT`
  fallback chain across `provideNgxSignalFormsConfig`,
  `provideNgxSignalFormControlPresets`, and `provideFieldLabels`. Static
  and reactive overloads, **nullish-only short-circuit** (preserves `''`,
  `0`, `false`), dev-mode `__source` tagging via the `CASCADING_SOURCE`
  symbol. Side-effect: `provideErrorMessages({ required: '' })` now
  preserves the empty string instead of silently falling through to the
  built-in default (this was a latent correctness bug — empty-string
  validator messages were being discarded).

- **`feat(core)`: `createErrorVisibility` consolidated factory**
  ([#27](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/27),
  PR [#33](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/33))
  — single factory that resolves strategy + submittedStatus + showErrors
  in one place. Powers `NgxSignalFormAutoAria` and is exposed as a
  public primitive (`createErrorVisibility`,
  `CreateErrorVisibilityOptions`, `ErrorVisibilityState`,
  `ErrorReadableState`) for consumers writing custom error UI on top
  of headless. Replaces ad-hoc combinations of
  `resolveStrategyFromContext` + `resolveSubmittedStatusFromContext` +
  `createShowErrorsComputed`.

- **`feat(toolkit)`: `NgxFieldIdentity` unified field-identity primitive**
  ([#26](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/26),
  PR [#34](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/34))
  — element-scoped service (`@Injectable({ providedIn: null })`) that
  centralises field-name resolution, unique ID generation
  (`errorId`, `warningId`), `aria-describedby` aggregation, control
  element reference, and visibility tracking. Provided by
  `NgxFormFieldWrapper` and consumed (optionally) by
  `NgxSignalFormAutoAria`. Two correctness wins:
  1. **`aria-invalid` clears when the bound control is hidden**
     (e.g. collapsed fieldset). Implemented via a `checkVisibility()`
     poll driven by the wrapper's existing `afterEveryRender` write
     phase — replaces an earlier `IntersectionObserver` design that
     leaked at directive destroy and reported stale state for
     scrolled-off-screen controls.
  2. **One-shot dev-mode warning** when the bound control has no `id`
     attribute and no explicit `fieldName` override is set, surfacing
     the `<label for>` / `aria-describedby` configuration gap early.
  - Public `BOUND_CONTROL_SELECTOR` / `findBoundControl` helper moved
    into `core/utilities/find-bound-control.ts` (re-exported from
    `form-field/form-field.utils.ts` for backward compat).

- **`feat(toolkit)`: consolidate field-tree traversal + delegate
  touched-on-submit to Angular**
  ([#24](https://github.com/ngx-signal-forms/ngx-signal-forms/issues/24),
  PR [#32](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/32))
  — every recursive walk over `FieldTree` (focus-first-invalid,
  mark-all-touched, error summary, debugger) now goes through one
  primitive. Public exports: `walkFieldTree`, `walkFieldTreeEntries`
  (lazy iterator), `walkFieldTreeIterable`, and a typed
  `InvalidFieldTreeError` (with `cause` support and a generic
  parameter). The walker reads `FieldTree` values **untracked** when
  detecting array-vs-object subtrees, so consumer `computed()` callers
  no longer subscribe to every leaf value signal.
  - **Mark-all-touched on submit** now delegates to native
    `submit({ ignoreValidators: 'all' })` instead of the toolkit
    walking the tree by hand. Lighter coupling to Angular's internal
    field traversal contract and same observable behaviour.

- **`build(types)`: exhaustiveness tests for `CONTROL_KIND_CAPABILITIES`**
  ([#29](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/29))
  — explicit unit tests that cross-check
  `CONTROL_KIND_CAPABILITIES` against
  `NGX_SIGNAL_FORM_CONTROL_KIND_VALUES`. The `satisfies`-based
  type-system guarantee already exists; these tests protect against
  accidental `as const` removal.

### 🐛 Fixes

- **`fix(toolkit)`: empty-string validator messages preserved**
  (PR [#31](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/31))
  — `resolveErrorMessage` now uses a nullish check (`if (msg != null)`)
  instead of a truthy check, so a deliberately empty validator message
  no longer silently falls through to the registry/default. Added a
  regression test in `error-messages.provider.spec.ts`.

- **`fix(toolkit)`: stable `aria-describedby` under shallow array equality**
  (PR [#34](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/34))
  — `NgxFieldIdentity._setHintIds` is now idempotent under shallow
  array equality, so `auto-aria`'s describedBy `computed()` no longer
  recomputes on stable hints.

- **`fix(toolkit)`: dual stable live regions on `NgxFormFieldNotification`**
  (PR [#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22))
  — split into `role="alert"` + `role="status"` shells so the role
  attribute never re-assigns at the same tick as content insertion
  (mirrors the `NgxFormFieldError` WCAG 4.1.3 fix shipped earlier).

- **`fix(toolkit)`: unnamed selection clusters get unique header ids**
  (PR [#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22))
  — wrapper no longer hard-codes `selection-group-label`, preventing
  duplicate-id ARIA misrouting when two unnamed clusters share a page.

- **`fix(toolkit)`: gate selection-cluster `aria-describedby` on visibility**
  (PR [#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22))
  — the warning-id reference no longer dangles when the error slot
  isn't mounted (reads `shouldShowErrors()` first).

- **`fix(toolkit)`: `[attr.id]` binding on empty notification shell**
  (PR [#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22))
  — empty notification shells no longer render `id="null"`; the
  fieldset's `aria-describedby` is gated on `showMessages()`.

- **`fix(toolkit)`: drop `::ng-deep` piercing in selection-cluster CSS**
  (PR [#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22))
  — wrapper no longer reaches into projected
  `[role='radiogroup']` / `[role='group']` / `<fieldset>` content. The
  responsibility shifts to consumer CSS (the demo's
  `.choice-group-field__options` already owns that layout). Public
  `--ngx-form-field-selection-group-*` tokens remain available as a
  drop-in normalisation recipe for consumers that want one.

- **`fix(toolkit)`: notification background adapts to colour scheme**
  (PR [#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22))
  — invalid selection-cluster background uses `color-mix()` against the
  current surface colour instead of a hard-coded light-mode red, and
  light-mode is now opt-in via an explicit `.light` class rather than
  forced via `prefers-color-scheme: dark` overrides.

- **`fix(toolkit)`: harden cascading-resolver against frozen objects**
  (PR [#31](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/31))
  — `attachSource()` now skips non-extensible values and swallows
  `defineProperty` failures so dev-mode source tagging can never break
  resolution.

- **`fix(toolkit)`: `focus-first-invalid` no longer swallows malformed
  field trees** (PR [#32](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/32))
  — removed a broad `InvalidFieldTreeError` swallow per #24 user story
  5; malformed `FieldTree` instances now surface the diagnostic instead
  of silently no-op-ing focus restoration.

### ♻️ Refactoring

- **`refactor(toolkit)!`: consolidate CSS theming tokens** (PR [#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22))
  — collapsed duplicated feedback / list / border tokens across
  error, notification, fieldset, hint, and character-count into a
  shared **Tier 2 ("shared feedback") layer**
  (`--ngx-signal-form-feedback-{font-size,line-height,margin-top,padding-horizontal,list-style,list-indent}`).
  Override one token to retune every feedback surface at once.
  - **`*-border` → `*-border-color`** across error, warning,
    notification, and fieldset grouped messages.
  - **`*-list-style-type` + `*-list-style-position` → `*-list-style`**
    shorthand.
  - **`*-padding-horizontal` shortcuts removed** in favour of the
    logical-property pair (`*-padding-inline-start` /
    `*-padding-inline-end`), or the shared
    `--ngx-signal-form-feedback-padding-horizontal`.
  - **`-override` escape hatches removed** —
    `--ngx-signal-form-error-{font-size,line-height}-override` were
    undocumented and the normal cascade already covers them.
  - **Legacy `--ngx-form-field-outline-*` aliases removed** — they were
    1:1 backward-compat shims; override the core
    `--ngx-form-field-*` tokens directly.
  - Fieldset stops cross-consuming wrapper `--ngx-form-field-color-*`
    tokens and now owns its own semantic colours that fall back to the
    wrapper's only as a last resort.
  - 13 dead `--ngx-signal-form-fieldset-selection-*` public CSS tokens
    dropped — they were documented as stable v1 API but never consumed
    (selection-cluster styling moved to the wrapper).
  - **See [`docs/MIGRATING_CSS_VARS.md`](../MIGRATING_CSS_VARS.md) for
    the full before/after table.**

- **`refactor(toolkit)!`: harden v1 API surface and a11y semantics**
  (PR [#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22))
  - `NgxFormFieldErrorPlacement` now shared by wrapper and fieldset.
  - `Ngx`-prefixed fieldset unions: `NgxFieldsetFeedbackAppearance`,
    `NgxFieldsetSurfaceTone`, `NgxFieldsetValidationSurface`.
  - Shared `NgxFormFieldListStyle` (old list-style names kept as
    deprecated aliases).
  - `NgxFormFieldNotification.errors` tightened to **signal-only**,
    matching `NgxFormFieldError`.
  - Fieldset: `describedByIds` downgraded to `protected`; host
    bindings now read validated resolved values.
  - Dropped the `--ngx-form-field-outline-min-height` back-compat shim
    to match the migration doc's "Removed" claim.

- **`refactor(toolkit)`: ship component styles as native CSS**
  (PR [#22](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/22))
  — every `packages/toolkit/**/*.scss` is now `.css`. Angular's
  partial-compilation output was already native CSS; the files only
  used Sass for `@use`/`@mixin` to share the Tier 2 feedback token
  defaults, which now lives in a colocated `feedback-tokens.css`.
  Native CSS nesting, `@media`, `color-mix()`, `:has()`, and
  `@supports (interpolate-size: allow-keywords)` blocks are preserved
  verbatim. **No public API or token name changes; consumers see
  identical computed styles.**
  - **Effective browser baseline pinned** in `THEMING.md`: Chrome /
    Edge 112, Safari 16.5, Firefox 121 (Jan 2024) — driven by native
    CSS nesting + `color-mix()` + `:has()` (the last one already
    required by `appearance="outline"`).

- **`refactor(headless)`: reusable bridge contract for `connectFieldState`**
  (PR [#30](https://github.com/ngx-signal-forms/ngx-signal-forms/pull/30))
  — `NgxHeadlessErrorState` exposes `connectFieldState()` and a
  public `resolvedSubmittedStatus` signal so styled hosts can bridge
  state without `hostDirectives` input forwarding.

### ⚠️ Breaking changes summary

| Area               | Change                                                             | Action                                                           |
| ------------------ | ------------------------------------------------------------------ | ---------------------------------------------------------------- |
| CSS tokens         | `*-border` → `*-border-color`                                      | Rename in stylesheets                                            |
| CSS tokens         | `*-list-style-type` + `*-list-style-position` → `*-list-style`     | Combine into one token                                           |
| CSS tokens         | `*-padding-horizontal` shortcuts removed                           | Use `*-padding-inline-{start,end}` or shared feedback token      |
| CSS tokens         | `--ngx-signal-form-error-{font-size,line-height}-override` removed | Set the non-`-override` token directly                           |
| CSS tokens         | `--ngx-form-field-outline-*` aliases removed                       | Use core `--ngx-form-field-*`                                    |
| CSS tokens         | 13 `--ngx-signal-form-fieldset-selection-*` tokens removed         | Style selection clusters on the wrapper                          |
| Fieldset default   | `errorPlacement: 'top'` → `'bottom'`                               | Pin `errorPlacement="top"` to keep prior DOM order               |
| Fieldset default   | `feedbackAppearance: 'auto'` resolves to `'notification'`          | Pass `feedbackAppearance="plain"` for the inline error treatment |
| Fieldset default   | `validationSurface: 'never'` (default), `'auto'` removed           | Opt in with `validationSurface="always"`                         |
| Notification API   | `errors` is signal-only                                            | Wrap static arrays in `signal([...])`                            |
| Validator messages | Empty string preserved (was discarded)                             | Audit `provideErrorMessages` overrides for intentional `''`      |

Full migration tables live in
[`docs/MIGRATING_CSS_VARS.md`](../MIGRATING_CSS_VARS.md) and
[`docs/MIGRATING_BETA_TO_V1.md`](../MIGRATING_BETA_TO_V1.md#fieldset-component--headless-directive).

### Suggested post-RC.7 watch items (not blockers)

- **Soak window on the new fieldset defaults** — the
  `errorPlacement: 'top' → 'bottom'` and
  `feedbackAppearance: 'auto' → 'notification'` flips are the most
  user-visible breaking changes in this RC. Worth a 1–2 week
  observation period before the stable cut to catch any
  consumer-visual regressions the migration guide doesn't anticipate.
- **Document `NgxFieldIdentity` for advanced consumers** — the new
  primitive is exposed via the wrapper today, but a short
  "writing a custom wrapper" recipe in `THEMING.md` or the assistive
  README would help consumers who want to skip the styled wrapper
  but keep the auto-aria wiring.
- **Consider promoting `NgxHeadlessNotification` examples** — the
  cross-surface specs prove parity with the styled component, but
  the assistive README doesn't yet show a "build your own
  notification card on top of the headless directive" recipe.

## Breaking-change cheatsheet (RC.6 → RC.7)

| Old                                                      | New                                                                                                               |
| -------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `--ngx-signal-form-error-border`                         | `--ngx-signal-form-error-border-color`                                                                            |
| `--ngx-signal-form-warning-border`                       | `--ngx-signal-form-warning-border-color`                                                                          |
| `--ngx-signal-form-notification-border`                  | `--ngx-signal-form-notification-border-color`                                                                     |
| `--ngx-signal-form-fieldset-*-border`                    | `--ngx-signal-form-fieldset-*-border-color`                                                                       |
| `--*-list-style-type` + `--*-list-style-position`        | `--*-list-style` (shorthand)                                                                                      |
| `--*-padding-horizontal` shortcuts                       | `--*-padding-inline-start` / `--*-padding-inline-end` (or shared `--ngx-signal-form-feedback-padding-horizontal`) |
| `--ngx-signal-form-error-font-size-override`             | (use `--ngx-signal-form-error-font-size` directly)                                                                |
| `--ngx-signal-form-error-line-height-override`           | (use `--ngx-signal-form-error-line-height` directly)                                                              |
| `--ngx-form-field-outline-*` aliases                     | core `--ngx-form-field-*` tokens                                                                                  |
| `--ngx-form-field-outline-min-height`                    | (use `--ngx-form-field-min-height`)                                                                               |
| `--ngx-signal-form-fieldset-selection-*` (×13)           | (selection styling moved to the wrapper; tokens dropped)                                                          |
| `<ngx-form-fieldset errorPlacement="…">` default `'top'` | default `'bottom'`                                                                                                |
| `<ngx-form-fieldset feedbackAppearance="…">` `'auto'`    | resolves to `'notification'`; `'auto'` accepted but treated as the new default                                    |
| `<ngx-form-fieldset validationSurface="auto">`           | removed; use `'never'` (default) or `'always'`                                                                    |
| `NgxFormFieldNotification.errors` accepting plain arrays | signal-only (wrap with `signal([...])`)                                                                           |
| Empty-string validator messages discarded                | preserved verbatim                                                                                                |

CSS classes (`.ngx-signal-form-field-wrapper__*`,
`.ngx-signal-form-fieldset--invalid`) and data attributes
(`data-ngx-signal-form-control-*`) are intentionally unchanged.
